💡 ¿Por qué está ocurriendo esto y cómo solucionarlo?
Hay tres escenarios muy comunes que causan exactamente este comportamiento en entornos macOS con PyAudio:

1. Conflicto de Concurrencia (Multihilo)
En el reporte se observa que tienes al menos dos hilos de ejecución en paralelo usando PortAudio: el Thread 4 (ThreadPoolExecutor-0_0) y el Thread 9 (ThreadPoolExecutor-0_1). Ambos están llamando activamente a la función ReadStream.

El problema: PortAudio no es completamente thread-safe por defecto si múltiples hilos intentan leer, escribir o cerrar el mismo flujo (stream) de manera simultánea sin una sincronización adecuada (como un Lock de exclusión mutua). Si un hilo intenta leer mientras otro está modificando o cerrando el flujo, se genera un puntero nulo en memoria.

Solución: Asegúrate de envolver las operaciones de lectura/escritura de PyAudio bajo un threading.Lock() para garantizar que solo un hilo interactúe con el stream a la vez.

2. Lectura de un Stream Cerrado o no Inicializado
El error ocurre cuando se invoca la lectura de datos, pero el flujo de audio subyacente pudo haber fallado al abrirse (por ejemplo, si el micrófono no dio permisos o cambió el dispositivo de entrada por defecto en la Mac) o fue cerrado abruptamente por otro proceso.

Solución: Verifica minuciosamente que el objeto de tu stream esté activo antes de llamar a leer, implementando bloques try/except para capturar cualquier anomalía antes de que escale a un error de sistema.

3. Incompatibilidad de Binarios con Python 3.14
Estás ejecutando una versión sumamente reciente o en desarrollo de Python (Python 3.14.5) instalada mediante Homebrew en una arquitectura Intel (X86-64). Las extensiones compiladas en C de PyAudio (_portaudio...so) suelen presentar fallos de memoria severos si no están perfectamente adaptadas o compiladas para la versión exacta del intérprete de Python que las invoca.

Solución: Si los pasos de sincronización de hilos no resuelven el problema, considera hacer un downgrade temporal de tu entorno de Python a una versión altamente estable (como Python 3.11 o 3.12) y reinstalar PyAudio desde allí para descartar incompatibilidades del binario nativo.
